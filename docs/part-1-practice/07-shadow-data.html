7  Shadow Data: um contêiner para qualquer tipo

As listas dos capítulos 5 e 6 guardavam int. Mas e se você quiser uma lista de double? De char *? De uma struct Produto? A resposta ingênua — copiar o arquivo e trocar int por double — leva a dez cópias quase idênticas para manter em sincronia. Em C não existe template como em C++ nem genéricos como em Java; precisamos de outra saída.

Este capítulo é sobre a estratégia que dá a você um contêiner genérico reutilizável: a Shadow Data (dados sombra). A ideia é separar o que a estrutura controla (o esqueleto de nós e ponteiros) do que ela guarda (o dado de negócio, de qualquer tipo) usando o ponteiro genérico void *. É o destaque do livro — e o momento em que todo o estudo de ponteiros começa a pagar dividendos.

Brief: um balcão de guarda-volumes: pessoas entregam objetos completamente diferentes (uma mala, um violão, uma caixa) e recebem um gancho numerado. O atendente organiza os ganchos por número sem se importar com o que há dentro de cada item. Ao lado, um rótulo: “a estrutura cuida dos ganchos (controle); o conteúdo é opaco (void*)”. Estilo lúdico à mão, paleta quente. Arte a comissionar — substituir este box.

7.1 A analogia: o guarda-volumes

Pense num guarda-volumes de museu. Você entrega qualquer coisa — uma mochila, um casaco, um capacete — e recebe uma ficha numerada. O sistema do guarda-volumes gerencia ganchos e números; ele não sabe (nem precisa saber) o que há dentro de cada item. Quando você volta com a ficha, ele devolve o que está pendurado naquele gancho.

A estrutura genérica funciona igual. Ela administra o controle — os nós, os ponteiros, o tamanho — e trata o dado de negócio como um pacote opaco, acessível só por um endereço (void *). Quem sabe o que há “dentro” é o código que usa a estrutura, não a estrutura. Essa separação entre o esqueleto (a sombra) e o dado real é o coração da Shadow Data.

7.2 O problema concreto: reescrever a lista por tipo

A lista do capítulo 6 amarrava o nó ao tipo: int value;. Para guardar outra coisa, seria preciso outro SNode, outro slist_push_front, outro tudo. Multiplique isso pelas estruturas do livro (pilha, fila, árvore…) e por cada tipo do seu programa: é uma explosão combinatória de código duplicado.

A Shadow Data corta o nó pela raiz: em vez de um int, o nó guarda um void *.

7.3 Do int ao void *: o nó-sombra

/*
 * generic_list.h — Lista encadeada GENÉRICA via void* (estratégia Shadow Data).
 *
 * Projeto:   PIESC — Estruturas de Dados e Algoritmos em C
 * Capítulo:  7 — Estratégia Shadow Data
 * Padrão:    C17 (ISO/IEC 9899:2018)
 * Build:     gcc/clang -std=c17 -Wall -Wextra -Wpedantic   (ver Makefile)
 */
#ifndef GENERIC_LIST_H
#define GENERIC_LIST_H

#include <stddef.h>  // size_t
#include <stdbool.h> // bool

// Um nó "sombra": guarda só CONTROLE — um ponteiro para o dado de negócio
// (que mora à parte) e o encadeamento. A lista não conhece o tipo do dado.
typedef struct GNode {
    void *data; // ponteiro para o dado real (a "sombra" aponta o negócio)
    struct GNode *next; // metadado de controle: o encadeamento
} GNode;

// Estilo qsort: ponteiros de função descrevem como tratar o tipo concreto.
typedef void (*FreeFn)(void *data);                 // como liberar um dado
typedef int (*CmpFn)(const void *a, const void *b); // <0, 0, >0 (igual qsort)
typedef void (*ApplyFn)(void *data, void *ctx);     // aplica algo a cada dado

typedef struct {
    GNode *head;
    size_t size;
    FreeFn free_fn; // libera cada dado ao destruir; NULL = a lista não é dona
} GList;

void glist_init(GList *list, FreeFn free_fn);
void glist_free(GList *list);
bool glist_push_front(GList *list, void *data);

void *glist_find(const GList *list, const void *key, CmpFn cmp);
void glist_foreach(const GList *list, ApplyFn fn, void *ctx);
size_t glist_size(const GList *list);

#endif // GENERIC_LIST_H

Olhe o GNode: ele não tem mais um valor, e sim um void *data — um endereço “sem tipo” para onde o dado realmente mora. O nó virou uma sombra: puro controle (o ponteiro para o dado e o next), sem nada do negócio. A Figura 7.1 mostra essa separação.

NotaIdeia central

Na lista do capítulo 6, o dado morava dentro do nó (int value). Na Shadow Data, o nó guarda apenas o endereço do dado, que mora à parte. O nó passa a ser um esqueleto reutilizável: trocar o tipo do dado não muda uma linha sequer da estrutura. O preço — você já deve estar farejando — é que a estrutura deixa de saber o que está guardando.

7.4 Ponteiros de função: ensinando o tipo à estrutura

Se a lista não conhece o tipo, como ela faz para comparar dois elementos numa busca, ou para liberar cada elemento ao ser destruída? Ela não faz — ela delega. O chamador entrega, junto, as funções que sabem lidar com o tipo concreto. É exatamente o padrão do qsort da biblioteca padrão, que recebe um ponteiro para a função de comparação.

No cabeçalho acima, três typedef definem esses contratos: FreeFn (como liberar um dado), CmpFn (como comparar dois, no mesmo formato <0, 0, >0 do qsort) e ApplyFn (o que fazer com cada dado ao percorrer). A estrutura guarda o free_fn e recebe os outros quando precisa.

7.5 O contêiner genérico

Criar e destruir mostram a Shadow Data em ação. Repare em glist_free: ele libera duas coisas distintas — o dado de negócio (via free_fn, se a lista for dona dele) e, separadamente, o nó-sombra (que é sempre nosso):

void glist_init(GList *list, FreeFn free_fn) {
    list->head = NULL;
    list->size = 0;
    list->free_fn = free_fn; // guarda como liberar o dado de negócio
}

// Destrói a lista. Repare na separação Shadow Data: liberamos o DADO (negócio,
// via free_fn — se a lista for dona) e depois o NÓ (controle, sempre nosso).
void glist_free(GList *list) {
    GNode *cur = list->head;
    while (cur != NULL) {
        GNode *next = cur->next;
        if (list->free_fn != NULL) {
            list->free_fn(cur->data); // o dado de negócio
        }
        free(cur); // o nó-sombra (controle)
        cur = next;
    }
    list->head = NULL;
    list->size = 0;
}

Inserir é como na lista comum, só que o nó recebe um void * em vez de um int:

// Insere no início. O nó é genérico: só guarda o ponteiro para o dado. O(1).
bool glist_push_front(GList *list, void *data) {
    GNode *node = malloc(sizeof(GNode));
    if (node == NULL) {
        return false;
    }
    node->data = data; // a sombra (nó) passa a apontar o dado real
    node->next = list->head;
    list->head = node;
    list->size += 1;
    return true;
}

Buscar é onde o ponteiro de função brilha: a lista varre os nós, mas quem decide se dois dados são “iguais” é o cmp fornecido pelo chamador — igualzinho ao qsort:

// Busca usando um comparador fornecido pelo chamador (igual ao qsort/bsearch).
// A lista não sabe comparar "o tipo"; ela delega ao CmpFn.
void *glist_find(const GList *list, const void *key, CmpFn cmp) {
    for (GNode *cur = list->head; cur != NULL; cur = cur->next) {
        if (cmp(cur->data, key) == 0) {
            return cur->data;
        }
    }
    return NULL;
}

E percorrer aplica uma função a cada dado, com um contexto opcional (útil para acumular um resultado, escrever num arquivo, etc.):

// Percorre aplicando uma função a cada dado, com um contexto opcional (ctx).
void glist_foreach(const GList *list, ApplyFn fn, void *ctx) {
    for (GNode *cur = list->head; cur != NULL; cur = cur->next) {
        fn(cur->data, ctx);
    }
}
AvisoCuidado: memory leak

Com void *, a posse (ownership) do dado vira uma decisão sua, explícita. A lista é dona dos dados e deve liberá-los? Então passe um free_fn (ex.: free). Os dados são emprestados (literais de string, objetos de outra estrutura)? Passe NULL e a lista não os toca. Errar isso causa leak (esquecer de liberar) ou double free (liberar algo que outro já liberou). O void * te dá poder e te cobra atenção: a estrutura não consegue mais te proteger automaticamente.

7.6 Uma lista, vários tipos

Aqui está a recompensa. A função main usa a mesma GList para guardar ints no heap (dos quais a lista é dona) e depois char * literais (dos quais não é) — sem uma linha de código de estrutura duplicada. As funções auxiliares (comparar, imprimir) são o que cada tipo traz de específico:

// Caixa um int no heap (a lista guardará o ponteiro para ele).
static int *box_int(int v) {
    int *p = malloc(sizeof(int));
    if (p != NULL) {
        *p = v;
    }
    return p;
}

// ApplyFn p/ int e p/ string; CmpFn p/ int (estilo qsort: <0, 0, >0).
static void print_int(void *data, void *ctx) {
    (void)ctx;
    printf(" %d", *(int *)data);
}
static void print_str(void *data, void *ctx) {
    (void)ctx;
    printf(" %s", (char *)data);
}
static int cmp_int(const void *a, const void *b) {
    int x = *(const int *)a;
    int y = *(const int *)b;
    return (x > y) - (x < y);
}
int main(void) {
    // A MESMA lista, dois tipos diferentes.
    // Lista de ints no heap: aqui a lista é DONA dos dados (free_fn = free).
    GList nums;
    glist_init(&nums, free);
    glist_push_front(&nums, box_int(10));
    glist_push_front(&nums, box_int(20));
    glist_push_front(&nums, box_int(30));

    printf("ints:");
    glist_foreach(&nums, print_int, NULL);

    int key = 20;
    int *hit = glist_find(&nums, &key, cmp_int);
    printf("\nbuscar 20 -> %s\n", hit != NULL ? "achou" : "nao achou");
    glist_free(&nums); // libera os nós E os ints (via free_fn)

    // Lista de strings literais: a lista NÃO é dona (free_fn = NULL).
    GList names;
    glist_init(&names, NULL);
    glist_push_front(&names, "ada");
    glist_push_front(&names, "linus");
    glist_push_front(&names, "grace");

    printf("nomes:");
    glist_foreach(&names, print_str, NULL);
    printf("\n");
    glist_free(&names); // libera só os nós; free_fn = NULL não toca nos dados

    return 0;
}

Saída do programa:

ints: 30 20 10
buscar 20 -> achou
nomes: grace linus ada

7.7 Os trade-offs do void *

ImportanteTrade-off: generalidade × type-safety × overhead

A Shadow Data compra máxima generalidade (uma implementação serve todo tipo, hoje e amanhã) ao preço de:

  • Perder a type-safety: o compilador não verifica mais nada. Guardar um int e ler como double, ou trocar a ordem dos argumentos de um cmp, compila sem reclamar e quebra em tempo de execução. Os casts são por sua conta e risco.
  • Pagar overhead: cada dado costuma exigir uma alocação separada no heap (mais pointer chasing, pior localidade de cache), e cada operação passa por uma chamada indireta via ponteiro de função, que o compilador não consegue inline.
  • Mais cerimônia: casts, funções auxiliares e a gestão manual de posse em todo ponto de uso.

Existe uma alternativa que faz o trade-off oposto — e todo projetista deveria conhecê-la.

7.8 O contraste: contêineres por macro (tempo de compilação)

E se, em vez de apagar o tipo em tempo de execução, nós gerássemos o código concreto para cada tipo em tempo de compilação? É o que um molde de macro faz: uma macro que, expandida, escreve a struct e as funções para o tipo que você pedir. O compilador volta a enxergar o tipo — então a type-safety e a velocidade voltam.

/*
 * generic_macro.h — Contêiner genérico em TEMPO DE COMPILAÇÃO (molde via
 * macro).
 *
 * Projeto:   PIESC — Estruturas de Dados e Algoritmos em C
 * Capítulo:  7 — Estratégia Shadow Data
 * Padrão:    C17 (ISO/IEC 9899:2018)
 */
#ifndef GENERIC_MACRO_H
#define GENERIC_MACRO_H

#include <stddef.h>  // size_t
#include <stdbool.h> // bool
#include <stdlib.h>  // realloc, free

// Molde: gera uma pilha type-safe e SEM overhead (dados inline) para o tipo T.
// Uso: DEFINE_STACK(int, IntStack) gera IntStack, IntStack_push/pop/free.
#define DEFINE_STACK(T, Name)                          \
    typedef struct {                                   \
        T *data;                                       \
        size_t size;                                   \
        size_t cap;                                    \
    } Name;                                            \
                                                       \
    static inline void Name##_init(Name *s) {          \
        s->data = NULL;                                \
        s->size = 0;                                   \
        s->cap = 0;                                    \
    }                                                  \
                                                       \
    static inline bool Name##_push(Name *s, T value) { \
        if (s->size == s->cap) {                       \
            size_t nc = s->cap ? s->cap * 2 : 4;       \
            T *nd = realloc(s->data, nc * sizeof(T));  \
            if (nd == NULL)                            \
                return false;                          \
            s->data = nd;                              \
            s->cap = nc;                               \
        }                                              \
        s->data[s->size++] = value;                    \
        return true;                                   \
    }                                                  \
                                                       \
    static inline bool Name##_pop(Name *s, T *out) {   \
        if (s->size == 0)                              \
            return false;                              \
        *out = s->data[--s->size];                     \
        return true;                                   \
    }                                                  \
                                                       \
    static inline void Name##_free(Name *s) {          \
        free(s->data);                                 \
        s->data = NULL;                                \
        s->size = 0;                                   \
        s->cap = 0;                                    \
    }

#endif // GENERIC_MACRO_H

A macro DEFINE_STACK(T, Name) usa o operador ## (token pasting, “colagem de tokens”) para montar nomes: Name##_push vira IntStack_push quando você instancia com DEFINE_STACK(int, IntStack). Cada uso gera um tipo completo, type-safe e com os dados inline (contíguos, como o array do capítulo 5 — sem void *, sem alocação por elemento):

#include "generic_macro.h"

#include <stdio.h>

// Uma linha instancia uma pilha de int completa, type-safe e sem void*.
DEFINE_STACK(int, IntStack)

int main(void) {
    IntStack s;
    IntStack_init(&s);
    IntStack_push(&s, 10);
    IntStack_push(&s, 20);
    IntStack_push(&s, 30);
    // IntStack_push(&s, 3.14);  // <- ERRO de compilação: o tipo é checado!

    int v;
    printf("IntStack (pop ate esvaziar):");
    while (IntStack_pop(&s, &v)) {
        printf(" %d", v);
    }
    printf("\n");

    IntStack_free(&s);
    return 0;
}

Saída do programa:

IntStack (pop ate esvaziar): 30 20 10

A linha comentada IntStack_push(&s, 3.14) não compilaria: o compilador sabe que a pilha é de int. Essa proteção é justamente o que o void * abre mão.

AvisoCuidado: o lado sombrio das macros

O poder das macros vem com armadilhas. Um erro de sintaxe dentro do molde gera mensagens do compilador apontando para a expansão, não para o seu código — crípticas e difíceis de rastrear. Não dá para stepar com o depurador dentro de uma macro. Cada instanciação duplica o código gerado (code bloat). E o pré-processador não tem noção de escopo nem de tipos: é substituição de texto, com todas as pegadinhas de parênteses e ponto-e-vírgula que isso implica. Por isso, para aprender, o void * é o melhor primeiro mestre; a macro é a ferramenta que vale reconhecer e usar com cuidado.

7.9 void * × macros: como escolher

Os dois caminhos para genéricos em C, fazendo trade-offs opostos.
Critério void * (Shadow Data) Macro (molde)
Verificação de tipo em execução (você) em compilação (compilador)
Overhead de tempo/espaço indireção + heap por item nenhum (inline)
Implementações geradas uma só, p/ todo tipo uma cópia por tipo
Mensagens de erro claras crípticas (na expansão)
Depuração normal difícil
Tipos em tempo de execução ✅ (lista heterogênea, plugins) ❌ (tipos fixos no build)
DicaPense como projetista

Use void * quando a generalidade em tempo de execução é o que importa: APIs de biblioteca que não conhecem os tipos do cliente, coleções heterogêneas, sistemas de plugin, ou quando uma única implementação compartilhada vale mais que o último ciclo de CPU (é por isso que o qsort da libc é void *). Use macros quando você tem poucos tipos conhecidos e quer a velocidade e a type-safety de código escrito à mão, sem reescrevê-lo (contêineres de alto desempenho como o klib). E, honestamente: na maior parte do código de aplicação, um array dinâmico concreto do tipo certo (capítulo 5) já resolve — alcance os genéricos quando a repetição doer de verdade.

NotaNo mundo real

A própria biblioteca padrão do C é Shadow Data: qsort e bsearch ordenam e buscam em qualquer array recebendo void *, o tamanho do elemento e um ponteiro para a função de comparação — o mesmo CmpFn deste capítulo. A GLib (base do GNOME) oferece GPtrArray/GList genéricos via gpointer (um void *). Do lado das macros, o klib (kvec.h, khash.h) entrega vetores e tabelas hash type-safe gerados por macro, e o sys/queue.h dos sistemas BSD/Linux dá listas e filas intrusivas inteiras feitas de macros.

7.10 Resumo

  • A Shadow Data separa o controle (o nó-esqueleto: ponteiros, tamanho) do dado de negócio (de qualquer tipo), guardando este último por um void *. Uma implementação serve a todos os tipos.
  • Como a estrutura deixa de conhecer o tipo, ela delega as operações específicas a ponteiros de função (cmp, free, apply) — o padrão do qsort.
  • O custo é perder a type-safety, pagar overhead (indireção, heap por item, pior cache) e gerenciar a posse dos dados à mão (free_fn ou NULL).
  • O contraponto são as macros-molde: geram código concreto por tipo, com type-safety e zero overhead — ao custo de erros crípticos, depuração difícil e code bloat.
  • Para aprender, prefira void * (concreto, depurável, é como a libc faz); saiba reconhecer as macros como a alternativa de tempo de compilação.
