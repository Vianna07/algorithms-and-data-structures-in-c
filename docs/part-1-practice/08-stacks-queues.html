8  Pilhas e Filas: o poder de restringir o acesso

As estruturas dos capítulos anteriores eram generalistas: você insere e remove onde quiser. Agora vamos no sentido oposto — estruturas que, de propósito, só deixam você mexer nas pontas. Parece um retrocesso, mas é o contrário: ao restringir o acesso, a estrutura ganha simplicidade, garante uma ordem de saída previsível e fica difícil de usar errado. Restringir é um recurso.

São duas, definidas pela ordem em que os elementos saem:

  • Pilha (stack): o último a entrar é o primeiro a sair — LIFO (Last In, First Out).
  • Fila (queue): o primeiro a entrar é o primeiro a sair — FIFO (First In, First Out).

E aqui está a lição central do capítulo: pilha e fila são interfaces (um contrato de operações), não implementações. Cada uma pode ser construída sobre os dois backings que já dominamos — o array dinâmico (cap. 5) e a lista encadeada (cap. 6). Vamos montar as duas de cada jeito e comparar.

8.1 Pilha (LIFO): a pilha de pratos

Pense numa pilha de pratos na pia. Você sempre coloca e tira do topo; o prato do fundo é o último a sair. Só existem três operações:

  • push: coloca um elemento no topo.
  • pop: tira o elemento do topo.
  • peek: espia o topo sem tirar.

Repare o que não existe: “inserir no meio”, “acessar o terceiro”. A pilha proíbe — e é essa proibição que a torna simples e segura. A Figura 8.1 ilustra as duas estruturas e por onde se entra e sai.

8.2 Dois backings para a mesma pilha

Sobre array dinâmico, a pilha é quase de graça: push é o append do capítulo 5 (escreve no fim, dobra a capacidade se encher) e pop só decrementa o tamanho. As duas operações mexem no fim do array — onde tudo é \(O(1)\).

// Pilha sobre array: empilhar é "append no fim"; desempilhar é "remove do fim".
bool as_push(ArrayStack *s, int value) {
    if (s->size == s->cap) { // cheio: dobra (como no cap. 5)
        size_t nc = s->cap ? s->cap * 2 : 4;
        int *nd = realloc(s->data, nc * sizeof(int));
        if (nd == NULL) {
            return false;
        }
        s->data = nd;
        s->cap = nc;
    }
    s->data[s->size++] = value;
    return true;
}

bool as_pop(ArrayStack *s, int *out) {
    if (s->size == 0) {
        return false;
    }
    *out = s->data[--s->size];
    return true;
}

Sobre lista encadeada, o topo é a cabeça da lista: push insere na cabeça, pop remove da cabeça — exatamente o caso \(O(1)\) do capítulo 6, sem deslocar nada e sem nunca realocar.

// Pilha sobre lista: o topo é a cabeça — empilhar/desempilhar é O(1) e nunca
// realoca, mas cada elemento custa um malloc e um ponteiro a mais.
bool ls_push(ListStack *s, int value) {
    LSNode *node = malloc(sizeof(LSNode));
    if (node == NULL) {
        return false;
    }
    node->value = value;
    node->next = s->top; // o novo nó vira o topo
    s->top = node;
    s->size++;
    return true;
}

bool ls_pop(ListStack *s, int *out) {
    if (s->top == NULL) {
        return false;
    }
    LSNode *top = s->top;
    *out = top->value;
    s->top = top->next; // o topo desce um
    free(top);
    s->size--;
    return true;
}
ImportanteTrade-off: array × lista como backing

As duas pilhas têm operações \(O(1)\), mas o caráter difere:

  • Array: memória contígua (ótima localidade de cache), sem ponteiro por elemento. Em troca, há realocações ocasionais \(O(n)\) (amortizadas) e pode sobrar capacidade ociosa.
  • Lista: nunca realoca e cresce de um em um, sem capacidade ociosa. Em troca, cada elemento custa um malloc e um ponteiro extra, e os nós ficam espalhados pela RAM (pior cache).

Para a maioria dos usos, a pilha sobre array vence — pela cache e por evitar um malloc por elemento. Prefira a lista quando não puder tolerar a pausa ocasional de uma realocação grande, ou quando os elementos já vivem em nós encadeados.

8.3 Fila (FIFO): a fila da padaria

Na fila da padaria, quem chega entra no fim; quem é atendido sai pela frente. As operações:

  • enqueue: põe um elemento no fim.
  • dequeue: tira o elemento da frente.

O detalhe que torna a fila mais interessante que a pilha: ela mexe nas duas pontas — entra numa, sai na outra. E é justamente isso que cria um problema quando tentamos usar um array comum.

8.3.1 Por que um array ingênuo não serve

Imagine uma fila sobre um array, com a frente no índice 0. Cada dequeue tira o elemento 0 e… precisa deslocar todos os outros uma casa para a esquerda — \(O(n)\) por remoção. Inaceitável. A saída elegante é não deslocar nada: deixar a frente “andar” pelo array e dar a volta quando chega ao fim. É o buffer circular (ring buffer).

8.4 A fila circular (ring buffer)

A ideia: um array de capacidade fixa e dois números — head (índice da frente) e count (quantos elementos há). O fim fica em (head + count) % cap. Quando um índice passaria do fim do array, o módulo o traz de volta ao começo. Nada se desloca; os índices é que circulam.

// O segredo do ring buffer: os índices "dão a volta" com o módulo (% cap).
// Guardamos head + count (não head + tail) p/ distinguir cheia de vazia.
bool rq_enqueue(RingQueue *q, int value) {
    if (q->count == q->cap) {
        return false; // cheia
    }
    size_t tail = (q->head + q->count) % q->cap; // 1º slot livre, com wrap
    q->data[tail] = value;
    q->count++;
    return true;
}

bool rq_dequeue(RingQueue *q, int *out) {
    if (q->count == 0) {
        return false; // vazia
    }
    *out = q->data[q->head];
    q->head = (q->head + 1) % q->cap; // a frente avança, com wrap
    q->count--;
    return true;
}

A Figura 8.2 mostra um buffer de capacidade 5 com a frente em [1]: ao avançar além de [4], o índice volta a [0].

DicaIdeia central

Por que guardar head + count em vez de head + tail (frente e fim)? Porque só com dois índices a condição head == tail é ambígua: vale tanto para fila vazia quanto cheia. Rastrear o count desfaz a ambiguidade de imediato. (A alternativa clássica é manter um slot sempre vazio para distinguir os dois casos — funciona, mas desperdiça uma posição.)

8.5 A fila sobre lista

Sobre lista encadeada, a fila é direta: guarde ponteiros para a frente (head) e o fim (tail). enqueue costura no tail, dequeue remove do head — ambos \(O(1)\), e a fila cresce sem limite de capacidade.

// Fila sobre lista: entra no fim (tail), sai pela frente (head). Os dois
// ponteiros tornam enqueue e dequeue O(1), sem limite de capacidade.
bool lq_enqueue(ListQueue *q, int value) {
    LQNode *node = malloc(sizeof(LQNode));
    if (node == NULL) {
        return false;
    }
    node->value = value;
    node->next = NULL;
    if (q->tail != NULL) {
        q->tail->next = node; // costura no fim
    } else {
        q->head = node; // fila estava vazia: node é frente e fim
    }
    q->tail = node;
    q->size++;
    return true;
}

bool lq_dequeue(ListQueue *q, int *out) {
    if (q->head == NULL) {
        return false;
    }
    LQNode *front = q->head;
    *out = front->value;
    q->head = front->next; // a frente avança
    if (q->head == NULL) {
        q->tail = NULL; // ficou vazia: zera a cauda também
    }
    free(front);
    q->size--;
    return true;
}

A diferença de projeto em relação ao ring buffer: a fila sobre lista nunca enche (cresce sob demanda), mas paga um malloc/free por elemento e perde a localidade de cache. O ring buffer tem capacidade fixa, mas zero alocações após a criação e memória contígua — por isso é o queridinho de sistemas de tempo real.

8.6 Vendo as quatro em ação

int main(void) {
    int v;

    // Pilha sobre array (LIFO): sai na ordem INVERSA da entrada.
    ArrayStack as;
    as_init(&as);
    for (int i = 1; i <= 3; i++) {
        as_push(&as, i * 10);
    }
    printf("pilha/array (LIFO):");
    while (as_pop(&as, &v)) {
        printf(" %d", v);
    }
    as_free(&as);

    // Pilha sobre lista (LIFO): MESMO comportamento, outro backing.
    ListStack ls;
    ls_init(&ls);
    for (int i = 1; i <= 3; i++) {
        ls_push(&ls, i * 10);
    }
    printf("\npilha/lista (LIFO):");
    while (ls_pop(&ls, &v)) {
        printf(" %d", v);
    }
    ls_free(&ls);

    // Fila sobre lista (FIFO): sai na ordem de CHEGADA.
    ListQueue lq;
    lq_init(&lq);
    for (int i = 1; i <= 3; i++) {
        lq_enqueue(&lq, i * 10);
    }
    printf("\nfila/lista  (FIFO):");
    while (lq_dequeue(&lq, &v)) {
        printf(" %d", v);
    }
    lq_free(&lq);

    // Fila circular (cap 3): o 40 "dá a volta" e ocupa o slot que o 10 deixou.
    RingQueue rq;
    rq_init(&rq, 3);
    rq_enqueue(&rq, 10);
    rq_enqueue(&rq, 20);
    rq_enqueue(&rq, 30); // cheia
    rq_dequeue(&rq, &v); // sai o 10, libera um slot
    rq_enqueue(&rq, 40); // 40 entra no slot do 10 (wrap)
    printf("\nfila circular     :");
    while (rq_dequeue(&rq, &v)) {
        printf(" %d", v); // 20 30 40
    }
    rq_free(&rq);

    printf("\n");
    return 0;
}

Saída do programa:

pilha/array (LIFO): 30 20 10
pilha/lista (LIFO): 30 20 10
fila/lista  (FIFO): 10 20 30
fila circular     : 20 30 40

As duas pilhas saem invertidas (LIFO); a fila sai na ordem de chegada (FIFO). Na fila circular, o 40 enfileirado após um dequeue ocupa o slot que o 10 liberou — o wrap em ação.

8.7 Veja acontecer

O buffer circular, passo a passo. Arraste para aplicar operações a um ring de capacidade 3. Veja head e o fim circularem e os slots serem reaproveitados quando os índices dão a volta (laranja = ocupado, cinza = livre):

viewof passo = Inputs.range([0, 9], { value: 3, step: 1, label: "operações aplicadas" })
ring_ops = [
  { op: "enqueue", v: 10 }, { op: "enqueue", v: 20 }, { op: "enqueue", v: 30 },
  { op: "dequeue" }, { op: "enqueue", v: 40 }, { op: "dequeue" },
  { op: "enqueue", v: 50 }, { op: "dequeue" }, { op: "dequeue" }
]
{
  const cap = 3;
  const buf = new Array(cap).fill(null);
  let head = 0, count = 0, last = "—";
  for (let i = 0; i < passo; i++) {
    const o = ring_ops[i];
    if (o.op === "enqueue") {
      if (count < cap) { buf[(head + count) % cap] = o.v; count++; last = `enqueue(${o.v})`; }
      else last = `enqueue(${o.v}) — cheia, rejeitada`;
    } else {
      if (count > 0) { last = `dequeue() → ${buf[head]}`; head = (head + 1) % cap; count--; }
      else last = "dequeue() — vazia";
    }
  }
  const tail = (head + count) % cap;
  const occupied = (s) => { for (let j = 0; j < count; j++) if ((head + j) % cap === s) return true; return false; };
  const cell = (s) => {
    const occ = occupied(s);
    const marks = [];
    if (count > 0 && s === head) marks.push("head");
    if (count < cap && s === tail) marks.push("fim");
    return html`<div style="display:flex;flex-direction:column;align-items:center;gap:3px">
      <div style="width:48px;height:48px;display:flex;align-items:center;justify-content:center;border-radius:8px;font:600 15px sans-serif;background:${occ ? "#F5793B" : "#E9E4DC"};color:${occ ? "#fff" : "#999"}">${occ ? buf[s] : "·"}</div>
      <div style="font:10px sans-serif;color:#777">[${s}]</div>
      <div style="font:10px sans-serif;color:#2A9D8F;height:12px">${marks.join(" · ")}</div>
    </div>`;
  };
  return html`<div>
    <div style="display:flex;gap:12px;align-items:flex-start">${Array.from({ length: cap }, (_, s) => cell(s))}</div>
    <div style="margin-top:8px;font:13px sans-serif;color:#264653">última operação: <b>${last}</b> &nbsp;·&nbsp; count = ${count}/${cap}</div>
  </div>`;
}

8.8 Pense como projetista

DicaPense como projetista

Escolha pilha quando a ordem natural do problema é “desfazer o mais recente primeiro”: histórico de undo, avaliação de expressões, a própria pilha de chamadas, backtracking. Escolha fila quando é “atender por ordem de chegada”: tarefas a processar, eventos, mensagens, busca em largura. Para o backing: array (ou ring buffer) é o padrão — rápido e cache-friendly; vá para a lista quando precisar crescer sem limite e sem pausas de realocação, ou quando os elementos já moram em nós. E não esqueça: ao restringir a interface a push/pop ou enqueue/dequeue, você documenta a intenção e blinda o código contra usos indevidos.

NotaNo mundo real

A pilha de chamadas (call stack) do seu programa é literalmente uma pilha: cada função chamada empilha um quadro com suas variáveis locais, que é desempilhado no return — e o stack overflow é ela estourando. Ring buffers estão em todo sistema que processa um fluxo contínuo: placas de áudio (o buffer entre o programa e o alto-falante), pilhas de rede (pacotes que chegam mais rápido do que são processados), logs de alta velocidade. E há a deque (double-ended queue): uma fila que aceita inserir e remover nas duas pontas, generalizando pilha e fila — é o que está por trás do deque do Python e do std::deque do C++.

8.9 Resumo

  • Pilha (LIFO) e fila (FIFO) são interfaces restritas: só se mexe nas pontas. A restrição é proposital — traz simplicidade, ordem previsível e segurança.
  • A mesma interface roda sobre dois backings: array dinâmico (cap. 5) ou lista encadeada (cap. 6). Todas as operações essenciais são \(O(1)\).
  • A fila sobre array exige o truque do buffer circular (% cap) para não pagar \(O(n)\) deslocando elementos a cada dequeue.
  • Array/ring: contíguo, cache-friendly, sem malloc por item, mas com capacidade (fixa ou amortizada). Lista: cresce sem limite e sem realocar, mas com ponteiro e alocação por elemento e pior cache.
