6  Listas encadeadas: quando o vizinho mora longe

No capítulo anterior, o array dinâmico nos deu acesso por índice em \(O(1)\) e uma memória compacta — ao custo de deslocar elementos para inserir no meio e de realocar e copiar tudo ao crescer. Agora vamos ao outro lado desse trade-off: uma estrutura em que inserir e remover é barato e nunca há realocação, mas onde perdemos o acesso direto por índice. É a lista encadeada.

A diferença de filosofia é radical. No array, os elementos são vizinhos obrigatórios na memória: o elemento 5 fica logo depois do 4. Na lista encadeada, cada elemento mora onde der no heap, e guarda no próprio corpo o endereço de onde está o próximo. Ninguém precisa ser vizinho de ninguém.

6.1 A analogia: a caça ao tesouro

Imagine uma caça ao tesouro. A primeira pista está na sua mão; ela não contém o tesouro, mas diz onde está a próxima pista. Você vai até lá, encontra a segunda pista, que aponta para a terceira, e assim por diante até uma pista que diz “acabou”.

Note duas consequências dessa brincadeira, que são exatamente as da lista encadeada:

  1. As pistas podem estar espalhadas pela cidade inteira. Não precisam ser vizinhas. Acrescentar uma pista nova no meio do percurso é fácil: basta reescrever um endereço. (Inserir é barato.)
  2. Para chegar à 7ª pista, você é obrigado a passar pelas 6 anteriores. Não dá para “pular” direto. (Acesso por índice é caro.)

Guarde esse par: inserção barata, acesso caro. É o espelho do array.

6.2 O nó: um dado e um endereço

A peça fundamental é o nó (node): uma struct que carrega o dado de negócio e um ponteiro para o próximo nó.

/*
 * singly_list.h — Lista simplesmente encadeada de inteiros.
 *
 * Projeto:   PIESC — Estruturas de Dados e Algoritmos em C
 * Capítulo:  6 — Listas encadeadas
 * Padrão:    C17 (ISO/IEC 9899:2018)
 * Build:     gcc/clang -std=c17 -Wall -Wextra -Wpedantic   (ver Makefile)
 */
#ifndef SINGLY_LIST_H
#define SINGLY_LIST_H

#include <stddef.h>  // size_t
#include <stdbool.h> // bool

typedef struct SNode {
    int value;          // dado de negócio
    struct SNode *next; // ponteiro para o próximo nó (NULL marca o fim)
} SNode;

typedef struct {
    SNode *head; // primeiro nó (NULL se a lista está vazia)
    size_t size; // quantos nós existem
} SList;

void slist_init(SList *list);
void slist_free(SList *list);

bool slist_push_front(SList *list, int value);                // O(1)
bool slist_insert_after(SList *list, SNode *node, int value); // O(1) dado o nó

SNode *slist_find(const SList *list, int value); // O(n)
bool slist_remove(SList *list, int value);       // O(n): remove a 1ª ocorrência
size_t slist_size(const SList *list);

#endif // SINGLY_LIST_H

Repare no campo next: é um ponteiro para outra struct do mesmo tipo (struct SNode *). É isso que “encadeia” a lista. A struct SList em si guarda só dois campos: o ponteiro para a cabeça (head, o primeiro nó) e o size. A Figura 6.1 mostra uma lista com três nós:

NotaIdeia central

No array, a posição de um elemento é a informação de ordem — o vizinho seguinte é o próximo endereço, automaticamente. Na lista encadeada, a ordem é dada explicitamente por ponteiros: cada nó diz quem vem depois dele. Isso liberta os nós da obrigação de serem contíguos — e é a origem de todas as vantagens e desvantagens que veremos.

6.3 Inserir no início: o caso barato — \(O(1)\)

Como a lista guarda a cabeça, inserir um nó no começo é trivial e não depende do tamanho: criamos o nó, apontamos seu next para a antiga cabeça, e movemos a cabeça para o nó novo.

// Insere no início: o caso barato da lista encadeada. O(1).
bool slist_push_front(SList *list, int value) {
    SNode *node = malloc(sizeof(SNode));
    if (node == NULL) {
        return false;
    }
    node->value = value;
    node->next = list->head; // o novo nó aponta para a antiga cabeça
    list->head = node;       // e passa a ser a cabeça
    list->size += 1;
    return true;
}

São duas religações de ponteiro e um malloc — custo constante, \(O(1)\), independentemente de a lista ter 3 ou 3 milhões de nós. Compare com o array: para inserir no início de um array, seria preciso empurrar todos os elementos uma casa para a direita — \(O(n)\). Aqui está a primeira vitória da lista.

Inserir depois de um nó que você já tem em mãos é igualmente barato:

// Insere logo depois de um nó já conhecido. Sem deslocar nada: O(1).
bool slist_insert_after(SList *list, SNode *node, int value) {
    SNode *fresh = malloc(sizeof(SNode));
    if (fresh == NULL) {
        return false;
    }
    fresh->value = value;
    fresh->next = node->next; // costura: fresh aponta para quem vinha depois
    node->next = fresh;       // e node passa a apontar para fresh
    list->size += 1;
    return true;
}

A ordem das duas linhas importa: primeiro fresh->next = node->next (o nó novo passa a apontar para quem vinha depois), só então node->next = fresh. Inverter isso perderia o resto da lista.

6.4 Remover: o truque do ponteiro para ponteiro

Remover exige primeiro achar o nó (varrendo desde a cabeça, \(O(n)\)) e depois religar o anterior diretamente ao seguinte, pulando o nó removido. O ponto delicado é o “anterior”: e se o nó a remover for a própria cabeça?

A solução elegante usa um ponteiro para ponteiro (SNode **), que aponta para o campo que aponta o nó atual — seja ele list->head ou o next de outro nó. Assim, religar é sempre a mesma linha, sem caso especial para a cabeça:

// Remove a primeira ocorrência. O(n) para achar, O(1) para religar.
// Truque do "ponteiro para ponteiro": evita tratar a cabeça como caso especial.
bool slist_remove(SList *list, int value) {
    SNode **link = &list->head; // endereço do ponteiro que aponta o nó atual
    while (*link != NULL) {
        SNode *cur = *link;
        if (cur->value == value) {
            *link = cur->next; // religa o anterior diretamente ao próximo
            free(cur);
            list->size -= 1;
            return true;
        }
        link = &cur->next; // avança para o ponteiro 'next' do nó atual
    }
    return false;
}

6.5 Por que indexar custa \(O(n)\) — e o cache não ajuda

Quer o elemento de índice k? Não há atalho: é preciso partir da cabeça e seguir next k vezes. Buscar por valor é igual — varrer até achar.

// Busca a primeira ocorrência de um valor. O(n): segue ponteiro a ponteiro.
SNode *slist_find(const SList *list, int value) {
    for (SNode *cur = list->head; cur != NULL; cur = cur->next) {
        if (cur->value == value) {
            return cur;
        }
    }
    return NULL;
}

Há um custo escondido além do Big-O, e ele é o reverso exato da vantagem do array. Como os nós foram alocados em momentos diferentes com malloc, eles estão espalhados pelo heap. Seguir um next é um salto para um endereço imprevisível — o processador não consegue “adivinhar” e pré-carregar o próximo nó no cache. É o chamado pointer chasing: cada passo pode ser uma falha de cache. No array contíguo, varrer é uma rajada sequencial que a CPU adora; na lista, é um ziguezague pela RAM.

DicaComplexidade (Big-O)

Na lista simplesmente encadeada:

  • Inserir no início ou depois de um nó conhecido: \(O(1)\) de tempo.
  • Buscar / acessar por índice: \(O(n)\) — sem acesso aleatório.
  • Remover um valor: \(O(n)\) para achar, \(O(1)\) para religar.
  • Espaço: \(O(n)\), mas com um custo fixo por elemento que o array não tem — o ponteiro next (8 bytes numa máquina de 64 bits) somado ao overhead de cada bloco do malloc. Guardar um int de 4 bytes pode custar 24 bytes ou mais.

6.6 Lista duplamente encadeada: o ponteiro extra

A lista simples só anda para a frente. Se você está num nó e quer o anterior, não há como — teria de varrer de novo desde a cabeça. A lista duplamente encadeada resolve isso dando a cada nó um segundo ponteiro, prev, além de guardar também a cauda (tail) da lista:

/*
 * doubly_list.h — Lista duplamente encadeada de inteiros (com cauda).
 *
 * Projeto:   PIESC — Estruturas de Dados e Algoritmos em C
 * Capítulo:  6 — Listas encadeadas
 * Padrão:    C17 (ISO/IEC 9899:2018)
 * Build:     gcc/clang -std=c17 -Wall -Wextra -Wpedantic   (ver Makefile)
 */
#ifndef DOUBLY_LIST_H
#define DOUBLY_LIST_H

#include <stddef.h>  // size_t
#include <stdbool.h> // bool

typedef struct DNode {
    int value;          // dado de negócio
    struct DNode *prev; // nó anterior (NULL na cabeça) — o ponteiro extra
    struct DNode *next; // próximo nó (NULL na cauda)
} DNode;

typedef struct {
    DNode *head; // primeiro nó
    DNode *tail; // último nó: torna o push_back O(1)
    size_t size;
} DList;

void dlist_init(DList *list);
void dlist_free(DList *list);

bool dlist_push_front(DList *list, int value); // O(1)
bool dlist_push_back(DList *list, int value);  // O(1) graças à cauda
void dlist_remove(DList *list, DNode *node);   // O(1) dado o nó

size_t dlist_size(const DList *list);

#endif // DOUBLY_LIST_H

A Figura 6.3 mostra a estrutura: setas azuis para a frente (next) e setas vermelhas para trás (prev); head e tail dão acesso \(O(1)\) às duas pontas.

O ganho concreto: com prev e tail, inserir no fim (push_back) é \(O(1)\) —

// Insere no fim em O(1) — só é possível porque guardamos a cauda (tail).
bool dlist_push_back(DList *list, int value) {
    DNode *node = malloc(sizeof(DNode));
    if (node == NULL) {
        return false;
    }
    node->value = value;
    node->next = NULL;       // será o novo último nó
    node->prev = list->tail; // costura para trás, à cauda atual

    if (list->tail != NULL) {
        list->tail->next = node; // a antiga cauda aponta para o novo nó
    } else {
        list->head = node; // lista estava vazia: node é cabeça e cauda
    }
    list->tail = node;
    list->size += 1;
    return true;
}

— e remover um nó que você já tem em mãos também vira \(O(1)\), porque o prev dá o anterior de graça, sem varrer:

// Remove um nó conhecido em O(1): o ponteiro 'prev' dispensa varrer a lista
// para achar o anterior (era o que tornava a remoção O(n) na lista simples).
void dlist_remove(DList *list, DNode *node) {
    if (node->prev != NULL) {
        node->prev->next = node->next;
    } else {
        list->head = node->next; // node era a cabeça
    }
    if (node->next != NULL) {
        node->next->prev = node->prev;
    } else {
        list->tail = node->prev; // node era a cauda
    }
    free(node);
    list->size -= 1;
}
ImportanteTrade-off: o ponteiro extra

A lista dupla compra navegação bidirecional e remoção \(O(1)\) por nó ao preço de mais memória (um ponteiro prev por nó, +8 bytes cada) e mais ponteiros para manter corretos em cada operação — toda inserção/remoção precisa costurar quatro ligações em vez de duas, e um deslize deixa a lista corrompida. Use lista dupla quando você realmente precisa andar para trás ou remover nós arbitrários com eficiência; caso contrário, a lista simples é mais leve e mais fácil de acertar.

6.7 Lista circular: a cobra que morde o próprio rabo

Uma variação útil: em vez de o último nó apontar para NULL, ele aponta de volta para a cabeça, formando um anel. Numa lista circular, não há “fim” — você pode girar indefinidamente. É a estrutura natural para rodízios (round-robin): distribuir tarefas entre N servidores em ciclo, alternar jogadores numa mesa, agendar processos. Costura-se igual à lista comum; muda apenas que tail->next volta a head (numa dupla circular, também head->prev = tail). O cuidado novo é a condição de parada: como não há NULL, percorrer exige contar size passos ou marcar o nó de partida — senão você gira para sempre.

6.8 Veja acontecer

push_front passo a passo. Arraste para empilhar nós no início e veja a cabeça sempre receber o mais recente — por isso a lista sai na ordem inversa da inserção. Cada passo é \(O(1)\), sem tocar nos nós já existentes:

viewof n_push = Inputs.range([0, 8], { value: 3, step: 1, label: "push_front aplicados" })
{
  const vals = Array.from({ length: n_push }, (_, i) => (n_push - i) * 10);
  const node = (v) => html`<span style="display:inline-flex;align-items:stretch;font:600 13px sans-serif">
      <span style="background:#F5793B;color:#fff;padding:6px 10px;border-radius:6px 0 0 6px">${v}</span>
      <span style="background:#E7F1EF;color:#2A9D8F;padding:6px 8px;border-radius:0 6px 6px 0">●</span>
    </span>`;
  return html`<div style="display:flex;align-items:center;gap:6px;flex-wrap:wrap">
    <b style="color:#2A9D8F">head</b><span>→</span>
    ${vals.flatMap((v) => [node(v), html`<span>→</span>`])}
    <span style="color:#999">NULL</span>
  </div>`;
}

6.9 Pense como projetista

DicaPense como projetista

A pergunta decisiva não é “lista ou array?”, mas “qual é o padrão de acesso?”. Se o seu código vive inserindo e removendo em pontos que já tem em mãos (uma fila de tarefas, uma lista de espera, nós que você guarda em outra estrutura), a lista encadeada brilha. Se vive lendo por índice e varrendo em ordem (a maior parte dos casos!), o array dinâmico quase sempre ganha — inclusive porque a localidade de cache faz a varredura voar. Na dúvida, comece com array; troque para lista quando o perfil de inserção/remoção provar que vale.

NotaNo mundo real

Quando uma lista encadeada ordenada precisa de busca rápida, entra a skip list: uma lista com “atalhos” probabilísticos em vários níveis que permitem pular trechos, alcançando busca em \(O(\log n)\) médio. É o que o Redis usa por baixo dos sorted sets. Mas vale a contraparte honesta: na prática, para percorrer e buscar, arrays costumam vencer listas mesmo quando o Big-O empata — a localidade de cache de um bloco contíguo supera o pointer chasing de nós espalhados pela RAM. A lista encadeada é uma ferramenta de nicho, não o padrão.

6.10 Resumo

  • A lista encadeada troca contiguidade por ponteiros explícitos: cada nó diz onde está o próximo, então os nós não precisam ser vizinhos na memória.
  • Isso inverte o perfil do array: inserir/remover em pontos conhecidos é \(O(1)\) e nunca há realocação, mas acesso por índice e busca são \(O(n)\), e não há localidade de cache.
  • A lista dupla adiciona prev (e tail): navegação nos dois sentidos e remoção \(O(1)\) por nó, ao custo de mais memória e mais ponteiros a manter.
  • A lista circular fecha o anel (tail->next = head), ideal para rodízios — com atenção à condição de parada.
  • Disciplina de memória: cada nó é um malloc com seu free; ao liberar a lista, guarde o next antes de liberar o nó atual.
