5  Array dinâmico: o vetor que cresce sozinho

Você já programou com arrays. Em C, eles têm um defeito incômodo: nascem com um tamanho fixo. Você decide int v[100]; e pronto — se chegarem 101 elementos, azar. Mas a vida real raramente avisa quantos itens virão. Como guardar uma lista que não sabemos o tamanho de antemão, sem desperdiçar memória nem estourar?

A resposta é o array dinâmico (também chamado ArrayList, vector ou growable array): um array contíguo que realoca a si mesmo quando enche. É a primeira estrutura do nosso caminho como projetistas — e a mais usada no mundo todo. Vamos construí-la do zero, desenhar cada byte que ela move, e aprender no caminho um conceito que assombra muita gente: a complexidade amortizada.

5.1 A analogia: a casa que dobra de tamanho

Imagine que você mora numa casa com 4 cômodos e todos estão ocupados. Chega mais um morador. Você tem duas opções:

  1. Mudar de casa toda vez que chega alguém — alugar uma casa com 5 cômodos, carregar todos os móveis, e repetir no próximo morador (6 cômodos), e no próximo (7)… Mudar é caro, e você mudaria o tempo todo.
  2. Mudar de casa, mas para o dobro — quando os 4 cômodos enchem, você aluga uma de 8. Carrega os móveis uma vez e ganha 4 cômodos livres de folga. Só vai mudar de novo quando esses 8 encherem — e aí pula para 16.

A segunda estratégia é o coração do array dinâmico. Mudar (copiar tudo) é caro, mas se você dobra o espaço a cada mudança, as mudanças ficam tão raras que, na média, cada novo morador custa quase nada. Guarde essa imagem: vamos formalizá-la na seção sobre complexidade amortizada.

5.2 A ideia central: size não é capacity

Toda a estrutura nasce de uma distinção que parece pequena e muda tudo:

NotaIdeia central

Um array dinâmico guarda dois números sobre si mesmo:

  • size (tamanho): quantos elementos de verdade existem agora.
  • capacity (capacidade): quantos elementos cabem no bloco que já reservamos no heap, antes de precisar realocar.

Sempre vale size ≤ capacity. O espaço entre os dois é a folga que torna as inserções baratas. Quando size alcança capacity, e só então, a estrutura cresce.

Visualmente, a struct mora na pilha e aponta para um bloco contíguo no heap. A Figura 5.1 mostra um array com três elementos (size = 3) num bloco com quatro lugares (capacity = 4):

A cor laranja marca os elementos (a faixa size); o cinza, a capacidade ociosa (a folga). Vamos manter essa convenção de cor no livro inteiro.

5.3 A struct e a interface

Começamos pelo contrato — o cabeçalho. Um projetista pensa primeiro em o que a estrutura oferece, depois em como ela cumpre. Para este capítulo, um array dinâmico de int basta; a versão genérica (com void*) chega no capítulo de Shadow Data.

/*
 * dynamic_array.h — Array dinâmico de inteiros (ArrayList).
 *
 * Projeto:   PIESC — Estruturas de Dados e Algoritmos em C
 * Capítulo:  5 — Array dinâmico
 * Padrão:    C17 (ISO/IEC 9899:2018)
 * Build:     gcc/clang -std=c17 -Wall -Wextra -Wpedantic   (ver Makefile)
 */
#ifndef DYNAMIC_ARRAY_H
#define DYNAMIC_ARRAY_H

#include <stddef.h>  // size_t
#include <stdbool.h> // bool

typedef struct {
    int *data;       // bloco contíguo de elementos, no heap
    size_t size;     // quantos elementos existem
    size_t capacity; // quantos cabem antes de realocar
} DynamicArray;

// Ciclo de vida.
bool da_init(DynamicArray *da, size_t initial_capacity);
void da_free(DynamicArray *da);

// Inserção e remoção no fim.
bool da_push(DynamicArray *da, int value);
bool da_pop(DynamicArray *da, int *out);

// Acesso por índice e tamanho.
int da_get(const DynamicArray *da, size_t index);
void da_set(DynamicArray *da, size_t index, int value);
size_t da_size(const DynamicArray *da);

#endif // DYNAMIC_ARRAY_H

As funções que podem falhar por falta de memória devolvem bool (true = ok). Esse hábito — admitir que malloc/realloc podem falhar — separa código de brinquedo de código de produção.

5.4 Nascer e morrer: da_init e da_free

A implementação mora em dynamic_array.c, que começa puxando o cabeçalho e as bibliotecas usadas pelos próximos trechos:

#include "dynamic_array.h"

#include <assert.h> // assert
#include <stdlib.h> // malloc, realloc, free
// Prepara um array vazio (capacidade inicial opcional).
// Devolve false só se a alocação inicial falhar.
bool da_init(DynamicArray *da, size_t initial_capacity) {
    da->data = NULL;
    da->size = 0;
    da->capacity = 0;

    if (initial_capacity > 0) {
        da->data = malloc(initial_capacity * sizeof(int));
        if (da->data == NULL) {
            return false; // sem memória: struct fica vazia, mas válida
        }
        da->capacity = initial_capacity;
    }
    return true;
}

// Libera o bloco e zera a struct (evita dangling pointer / double free).
void da_free(DynamicArray *da) {
    free(da->data); // free(NULL) é seguro
    da->data = NULL;
    da->size = 0;
    da->capacity = 0;
}
AvisoCuidado: memory leak

A regra de ouro deste livro: cada malloc/calloc/realloc tem exatamente um free. Aqui, todo o bloco vive em da->data; um único free no da_free quita a dívida. Zerar da->data para NULL depois do free evita dois bugs clássicos: o double free (liberar duas vezes) e o dangling pointer (usar um ponteiro que já aponta para memória devolvida).

5.5 O coração: da_push e a realocação

Inserir no fim é o caso comum. A lógica tem duas partes: se está cheio, cresce; depois, grava.

// Cresce o bloco dobrando a capacidade. Detalhe interno (static).
static bool da_grow(DynamicArray *da) {
    size_t new_capacity = (da->capacity == 0) ? 1 : da->capacity * 2;

    int *new_data = realloc(da->data, new_capacity * sizeof(int));
    if (new_data == NULL) {
        return false; // realloc falhou; o bloco antigo continua válido
    }

    da->data = new_data;
    da->capacity = new_capacity;
    return true;
}

// Insere um valor no fim. O(1) amortizado.
bool da_push(DynamicArray *da, int value) {
    if (da->size == da->capacity) { // sem folga?
        if (!da_grow(da)) {
            return false; // falhou ao crescer: nada é perdido
        }
    }
    da->data[da->size] = value; // grava na primeira posição livre
    da->size += 1;
    return true;
}

Três detalhes desse trecho merecem atenção:

  1. A regra de crescimento: dobrar. Partindo de uma capacidade vazia, o primeiro grow reserva 1; depois 2, 4, 8, 16… A próxima seção justifica por que dobrar (e não somar uma quantia fixa).
  2. realloc pode mover o bloco inteiro para outro endereço se não houver espaço contíguo logo após o atual. Quando isso acontece, ele copia os bytes antigos para o novo lugar — é o custo \(O(n)\) da realocação.
  3. Detalhe que evita vazamento: guardamos o retorno num ponteiro temporário (new_data). Se realloc falha, ele devolve NULL mas não libera o bloco antigo — que continua acessível por da->data. Se tivéssemos escrito da->data = realloc(da->data, ...), um NULL apagaria a única referência ao bloco e o vazaríamos.

5.6 Por que dobrar? A complexidade amortizada

Aqui está a pergunta que confunde muita gente: se uma realocação custa \(O(n)\) (copiar tudo), como podemos dizer que da_push é \(O(1)\)? A resposta é a análise amortizada — olhar o custo médio por operação ao longo de muitas operações, e não o pior caso isolado.

Vamos inserir \(n\) elementos do zero, dobrando a capacidade (\(1, 2, 4, \dots, n\)). As realocações copiam, no total:

\[ \underbrace{1 + 2 + 4 + 8 + \cdots + \tfrac{n}{2}}_{\text{cópias em cada realocação}} \;=\; n - 1 \;<\; n . \]

É uma soma geométrica: cada termo é a metade do seguinte, então o total não passa de \(n\), e não de \(n \log n\) como a intuição às vezes sugere. Somando as \(n\) gravações em si (cada uma \(O(1)\)), o custo de inserir \(n\) elementos é \(O(n)\) no total. Divida por \(n\) operações:

DicaComplexidade (Big-O)

da_push é \(O(1)\) amortizado. Inserir \(n\) elementos custa \(O(n)\) no total, logo \(O(1)\) em média por inserção — mesmo que uma inserção isolada possa custar \(O(n)\) quando dispara a realocação. “Amortizado” significa exatamente isto: o custo raro e caro se dilui sobre as muitas operações baratas que o cercam.

E por que dobrar, e não crescer de 1 em 1? Crescer +1 a cada vez forçaria uma cópia em toda inserção: \(1 + 2 + 3 + \cdots + n = O(n^2)\) no total, ou \(O(n)\) por push. Dobrar transforma esse \(O(n)\) em \(O(1)\) amortizado. A folga é o que compra a velocidade.

5.7 Acesso \(O(1)\) e o contraste central do livro

O array dinâmico tem um superpoder que vamos comparar com todas as estruturas seguintes: o acesso por índice em tempo constante.

// Lê o elemento em index. O(1).
int da_get(const DynamicArray *da, size_t index) {
    assert(index < da->size); // índice fora do tamanho = erro de programação
    return da->data[index];   // endereço = data + index*sizeof(int)
}

// Sobrescreve o elemento em index. O(1).
void da_set(DynamicArray *da, size_t index, int value) {
    assert(index < da->size);
    da->data[index] = value;
}

Por que \(O(1)\)? Porque o bloco é contíguo. O endereço do elemento index é simplesmente data + index * sizeof(int) — uma multiplicação e uma soma, sem percorrer nada. Não importa se você quer o elemento 0 ou o 1.000.000: o custo é o mesmo.

Há um segundo ganho, invisível no Big-O mas enorme na prática: a localidade de cache. Como os elementos são vizinhos na memória, quando a CPU busca um, ela já traz os vizinhos para o cache. Varrer um array dinâmico voa. Esse é o contraste que vai nos acompanhar:

ImportanteTrade-off: contíguo × encadeado
  • Array dinâmico (contíguo): acesso por índice \(O(1)\), localidade de cache excelente, memória compacta. Mas inserir/remover no meio custa \(O(n)\) (é preciso deslocar elementos), e crescer exige realocar e copiar.
  • Lista encadeada (próximo capítulo): inserir/remover é \(O(1)\) se você já está no ponto certo, sem realocação. Mas o acesso por índice vira \(O(n)\) (é preciso seguir ponteiro a ponteiro), e cada elemento gasta memória extra com ponteiros, espalhados pela RAM — péssima localidade de cache.

Não existe “melhor”: existe adequado ao uso. Acessa muito por índice e varre sequências? Contíguo. Insere e remove muito no meio, sem precisar de índice? Encadeado. Vamos repetir essa decisão em cada estrutura.

5.8 Remover do fim e o porquê de não encolher

// Remove o último elemento; copia-o para *out se out != NULL. O(1).
bool da_pop(DynamicArray *da, int *out) {
    if (da->size == 0) {
        return false; // nada a remover
    }
    da->size -= 1;
    if (out != NULL) {
        *out = da->data[da->size];
    }
    return true;
}

size_t da_size(const DynamicArray *da) {
    return da->size;
}

Repare que da_pop não devolve memória — apenas diminui size. A capacidade fica como estava, pronta para futuras inserções. Encolher (chamar realloc para um bloco menor) é possível, mas costuma ser má ideia: se o tamanho oscila perto de um limite, você entraria num vai-e-vem de realocações (thrashing). A heurística usual é encolher só quando size cai abaixo de um quarto da capacidade — mas, para a maioria dos usos, simplesmente não encolher é a escolha certa.

5.8.1 Vendo tudo funcionar

#include "dynamic_array.h"

#include <stdio.h>

int main(void) {
    DynamicArray numbers;
    da_init(&numbers, 2); // começa pequeno de propósito

    for (int i = 1; i <= 6; i++) {
        da_push(&numbers, i * 10); // força os crescimentos: 2 -> 4 -> 8
    }

    printf("size = %zu, capacity = %zu\n", da_size(&numbers), numbers.capacity);
    for (size_t i = 0; i < da_size(&numbers); i++) {
        printf("numbers[%zu] = %d\n", i, da_get(&numbers, i));
    }

    da_free(&numbers); // todo malloc tem seu free
    return 0;
}

Saída do programa:

size = 6, capacity = 8
numbers[0] = 10
numbers[1] = 20
numbers[2] = 30
numbers[3] = 40
numbers[4] = 50
numbers[5] = 60

Seis elementos couberam num bloco de capacidade 8: a estrutura cresceu 2 → 4 → 8 sozinha, sem o programador se preocupar.

5.10 Pense como projetista

DicaPense como projetista

O array dinâmico deve ser sua escolha padrão para uma coleção sequencial — o “até prova em contrário”. Ele é compacto, rapidíssimo para varrer e acessar por índice, e amortiza bem o crescimento. Só troque por outra estrutura quando o perfil de uso pedir: muita inserção/remoção no meio (pense em lista encadeada), busca por chave (tabela hash, fora do nosso escopo), ou ordem por prioridade (heap, capítulo 10). Comece simples; complique só quando os números exigirem.

NotaNo mundo real

Você raramente vai escrever um da_push na vida profissional — porque as linguagens já trazem o seu. O std::vector do C++, o ArrayList do Java e a list do Python são, por dentro, exatamente isto: um array dinâmico que dobra (ou cresce por um fator parecido) a capacidade ao encher. Quando você faz lista.append(x) em Python e ele “simplesmente funciona”, é o da_grow deste capítulo rodando lá no fundo, em C.

5.11 Resumo

  • Um array dinâmico é um bloco contíguo no heap que cresce sozinho via realloc, distinguindo size (o que existe) de capacity (o que cabe).
  • A estratégia de dobrar a capacidade torna a inserção no fim \(O(1)\) amortizado, mesmo com realocações \(O(n)\) ocasionais.
  • O ganho central é o acesso por índice \(O(1)\) e a localidade de cache; o custo é o \(O(n)\) para inserir/remover no meio. Esse é o trade-off contíguo × encadeado que guia o resto do livro.
  • Disciplina de memória: cada malloc/realloc tem seu free; proteja o ponteiro antigo numa variável temporária ao realocar.
