2  Gerenciamento de memória: a pilha e o heap

No capítulo 1, vimos que um ponteiro guarda um endereço. Mas de onde vêm esses endereços? Toda a memória de um programa se divide em duas regiões com regras opostas: a pilha (stack), automática e efêmera, e o heap, manual e duradouro. Cada malloc da Parte da prática pediu memória ao heap, e cada free pagou a conta. Este capítulo é sobre essa conta — o ciclo de vida de um bloco — e sobre os dois fantasmas que assombram todo programador C: o vazamento (leak) e o ponteiro pendurado (dangling).

2.1 As duas regiões: pilha e heap

Quando um programa roda, sua memória tem (entre outras) duas áreas que nos interessam:

  • A pilha (stack) guarda as variáveis locais e os “quadros” de cada chamada de função. É automática: ao entrar numa função, suas variáveis nascem; ao retornar, morrem sozinhas. É rápida, mas efêmera e limitada — e você não a controla diretamente.
  • O heap é uma área grande de onde você pede blocos sob demanda com malloc. É manual: o bloco vive até você mandar liberá-lo com free. Flexível e duradouro, mas com uma responsabilidade que é toda sua.

A Figura 2.1 mostra as duas convivendo: uma variável local p, na pilha, guardando o endereço de um bloco que vive no heap.

AvisoCuidado: não devolva o efêmero

Como as variáveis locais morrem no return, nunca devolva o endereço de uma variável local:

int *errado(void) {
    int x = 42;
    return &x;   // ❌ x morre aqui; o endereço devolvido é lixo
}

Para um valor sobreviver à função, ele precisa morar no heap (malloc) — não na pilha.

2.2 O ciclo de vida de um bloco: malloc e free

Trabalhar com o heap é seguir um ciclo de quatro passos: pedir, usar, devolver e zerar. O malloc reserva um bloco e devolve seu endereço (ou NULL se faltar memória); o free o devolve ao sistema.

// O ciclo de vida de um bloco no heap: pedir, usar, devolver — e zerar.
static void lifecycle(void) {
    int *a = malloc(3 * sizeof(int)); // 1) PEDIR memória
    if (a == NULL) {
        return; // malloc pode falhar (sem memória): sempre cheque
    }
    a[0] = 10; // 2) USAR
    a[1] = 20;
    a[2] = 30;
    printf("a = [%d, %d, %d]\n", a[0], a[1], a[2]);

    free(a);  // 3) DEVOLVER ao sistema
    a = NULL; // 4) ZERAR o ponteiro (evita dangling e double free)
}
DicaIdeia central

Cada malloc tem exatamente um free. É a regra de ouro do livro inteiro. Quem aloca é dono do bloco e responsável por liberá-lo — uma vez só, no momento certo. Zerar o ponteiro (a = NULL) após o free é o detalhe que evita os dois bugs deste capítulo de uma só tacada.

2.3 A família: calloc e realloc

Além do malloc, dois primos completam a família:

  • calloc(n, tamanho) aloca n elementos e os zera — conveniente para arrays que devem começar limpos (em malloc, o conteúdo é lixo).
// calloc: igual a malloc, mas ZERA a memória. Ideal para começar em zero.
static void calloc_demo(void) {
    int *zeros = calloc(4, sizeof(int)); // 4 ints, todos 0
    if (zeros == NULL) {
        return;
    }
    printf("calloc: [%d, %d, %d, %d]\n", zeros[0], zeros[1], zeros[2],
           zeros[3]);
    free(zeros);
}
  • realloc(ptr, novo_tamanho) redimensiona um bloco existente, preservando o conteúdo. Ele pode mover o bloco para outro endereço (e por isso devolve o novo ponteiro) — foi o motor do array dinâmico do capítulo 5.
// realloc: redimensiona um bloco preservando o conteúdo — e pode MOVÊ-LO.
static void realloc_demo(void) {
    int *v = malloc(2 * sizeof(int));
    if (v == NULL) {
        return;
    }
    v[0] = 1;
    v[1] = 2;

    int *bigger = realloc(v, 4 * sizeof(int)); // cresce de 2 para 4 ints
    if (bigger == NULL) {
        free(v); // realloc falhou: o bloco antigo (v) ainda é válido; libere-o
        return;
    }
    v = bigger; // só sobrescreve v DEPOIS de confirmar o sucesso
    v[2] = 3;
    v[3] = 4;
    printf("realloc: [%d, %d, %d, %d]\n", v[0], v[1], v[2], v[3]);
    free(v);
}
AvisoCuidado: o ponteiro temporário do realloc

Repare que guardamos o retorno do realloc numa variável nova (bigger) e só sobrescrevemos v após confirmar que não deu NULL. Escrever v = realloc(v, ...) direto é uma armadilha: se realloc falha e devolve NULL, você perde a única referência ao bloco antigo — que ainda existe — e o vaza.

2.4 Fantasma 1: o vazamento (memory leak)

Um vazamento acontece quando você perde a única referência a um bloco sem liberá-lo: o bloco continua reservado, mas ninguém consegue mais alcançá-lo para chamar free. Ele fica perdido até o programa terminar. Em programas longos (servidores, jogos), vazamentos se acumulam e consomem a memória até travar.

void vaza(void) {
    int *p = malloc(100);
    p = malloc(50);   // ❌ perdeu o 1º bloco: 100 bytes vazados para sempre
    free(p);          // libera só o 2º; o 1º virou um fantasma
}

A Figura 2.2 mostra o bloco órfão: p foi reapontado para o segundo bloco, e o primeiro ficou sem ninguém apontando para ele.

2.5 Fantasma 2: o ponteiro pendurado (dangling)

O oposto do vazamento: aqui você libera o bloco, mas continua usando o ponteiro que apontava para ele. Esse ponteiro está pendurado — aponta para memória que já foi devolvida e pode ser reusada por outra alocação a qualquer momento. Usá-lo é comportamento indefinido: às vezes “funciona”, às vezes corrompe dados, às vezes trava.

int *p = malloc(sizeof(int));
*p = 42;
free(p);            // o bloco volta ao sistema
printf("%d", *p);   // ❌ use-after-free: p está pendurado
free(p);            // ❌ double free: liberar de novo corrompe o heap

A Figura 2.3 mostra o estado perigoso: p ainda guarda o endereço, mas a casa já não é sua.

DicaA cura dos dois fantasmas

Os dois bugs têm o mesmo antídoto, que aplicamos no livro inteiro: libere no momento certo e zere o ponteiro logo depois (free(p); p = NULL;). Um ponteiro NULL não vaza (você ainda sabe que não há nada lá) e não fica pendurado (free(NULL) é seguro, e *NULL falha na hora, num erro fácil de achar, em vez de corromper silenciosamente).

2.6 Vendo funcionar

As três demonstrações de uso correto produzem:

a = [10, 20, 30]
calloc: [0, 0, 0, 0]
realloc: [1, 2, 3, 4]

O ciclo de vida, passo a passo. Avance e veja o bloco no heap nascer com malloc, ser usado, e voltar ao sistema com free — e por que zerar p no fim fecha a porta para o ponteiro pendurado:

2.7 A ferramenta: valgrind

NotaDetecte os fantasmas com o valgrind

Vazamentos e ponteiros pendurados são silenciosos — o programa pode parecer funcionar. O valgrind os caça para você: ele roda o programa numa máquina virtual que vigia cada malloc/free e cada acesso. valgrind --leak-check=full ./programa aponta exatamente onde um bloco vazou ou onde houve um use-after-free. Todo o código C deste livro passa pelo valgrind sem um único erro — e o seu também deveria.

2.8 Pense como projetista

DicaPense como projetista

A pergunta mais importante sobre qualquer bloco do heap é: quem é o dono? — isto é, quem tem a responsabilidade de chamá-lo free, e quando. Defina isso antes de alocar. Padrões que ajudam: alocar e liberar na mesma camada (a função que cria também destrói, como os pares da_init/da_free do livro); preferir a pilha quando o tempo de vida é claramente o do escopo (mais simples e impossível de vazar); e desconfiar de qualquer malloc cujo free correspondente você não consiga apontar com o dedo.

NotaNo mundo real

A dor de gerenciar memória à mão moldou as linguagens modernas. Java, Python e Go trazem um coletor de lixo (garbage collector) que rastreia referências e libera o que ninguém mais alcança — adeus vazamentos (quase), ao custo de pausas e de menos controle. C++ inventou o RAII (liberar no destrutor, automaticamente no fim do escopo) e os smart pointers. Rust vai além, com um sistema de posse (ownership) verificado em compilação que torna vazamentos e dangling praticamente impossíveis — uma formalização exata daquela pergunta “quem é o dono?”. Entender o heap em C é entender o problema que todas elas tentam resolver.

2.9 Resumo

  • A memória de um programa tem a pilha (variáveis locais, automática, some no fim do escopo) e o heap (blocos sob demanda, manual, vive até o free).
  • Nunca devolva o endereço de uma variável local — ela mora na pilha e morre no return.
  • O ciclo de um bloco é pedir → usar → devolver → zerar: malloc, usar, free, p = NULL. Cada malloc tem um free.
  • calloc zera; realloc redimensiona e pode mover (guarde o retorno num ponteiro temporário antes de confiar nele).
  • Vazamento: perdeu a referência sem liberar. Dangling: usou o ponteiro depois do free. A cura comum: liberar na hora certa e zerar o ponteiro. O valgrind caça os dois.
