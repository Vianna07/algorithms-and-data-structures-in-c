1  Ponteiros: o endereço é o dado

Bem-vindo aos fundamentos. Se você leu a Parte da prática, viu ponteiros em ação o tempo todo — node->next, int *out, void *data. Aqui destrinchamos a ideia única que sustenta tudo aquilo, e que é o conceito mais importante do C: o ponteiro. Quem entende ponteiros entende listas, árvores, malloc, void* e por que C é tão poderoso (e tão perigoso). Tudo se resume a uma distinção: o valor de uma variável versus o endereço onde ela mora.

1.1 A analogia: a rua de endereços

Pense na memória do computador como uma rua imensa de casas numeradas. Cada casa é um byte, e cada uma tem um endereço único (um número). Quando você declara int x = 42;, o compilador reserva uma casa (na verdade, quatro bytes seguidos) num certo endereço e guarda ali o valor 42.

Um ponteiro é um papelzinho onde você anota o endereço de uma casa. O papel não é a casa nem o móvel lá dentro — é só a localização. Com ele, você consegue chegar à casa e mexer no que há dentro. Essa é a essência: ponteiro guarda endereço; pelo endereço, chega-se ao valor.

1.2 Endereço vs. valor: & e *

Dois operadores fazem a ponte entre os dois mundos:

  • &x — “o endereço de x” (pega o número da casa).
  • *p — “o valor apontado por p” (entra na casa e olha/mexe no móvel).
// Endereço vs. valor: `&` pega o endereço; `*` acessa o valor apontado.
static void basics(void) {
    int x = 42;
    int *p = &x; // p guarda o ENDEREÇO de x (p "aponta para" x)

    printf("x        = %d\n", x);  // o valor de x
    printf("*p       = %d\n", *p); // o valor APONTADO por p (o mesmo de x)

    *p = 99;                      // escreve em x ATRAVÉS do ponteiro
    printf("x depois = %d\n", x); // 99: mudamos x sem citar o nome x
}

A Figura 1.1 desenha o estado depois de int *p = &x;: p guarda o endereço de x, e a seta representa o “aponta para”.

NotaIdeia central

x e p são duas variáveis distintas, em casas distintas. x guarda um valor (42); p guarda um endereço (o de x). & vai do valor para o endereço; * vai do endereço de volta ao valor. Confundir os dois é a fonte de metade dos bugs em C — e dominá-los é metade do domínio da linguagem.

1.3 Mudar à distância

O detalhe que torna ponteiros indispensáveis: por *p você não só lê como também escreve na casa apontada. No código acima, *p = 99; mudou x sem mencionar o nome x. Foi assim que, na prática, toda função que devolvia um resultado por um parâmetro int *out funcionou: ela escrevia na casa do chamador. É também como uma função consegue alterar uma variável de quem a chamou — em C, os argumentos são cópias, então passar o endereço é a única forma de deixar a função tocar no original.

1.4 Aritmética de ponteiros

Ponteiros e arrays são íntimos. O nome de um array “decai” no endereço do seu primeiro elemento, e somar a um ponteiro anda em elementos, não em bytes:

// Aritmética de ponteiros: conta em ELEMENTOS, não em bytes.
static void arithmetic(void) {
    int v[4] = {10, 20, 30, 40};
    int *p = v; // o nome do array "decai" no endereço de v[0]

    printf("*p       = %d\n", *p);       // 10
    printf("*(p + 2) = %d\n", *(p + 2)); // 30: avança 2 ELEMENTOS

    // p + 1 pula sizeof(int) bytes; a distância em bytes prova isso:
    printf("bytes(v[1] - v[0]) = %td\n", (char *)&v[1] - (char *)&v[0]);
}

p + 2 não pula 2 bytes — pula 2 ints, ou seja, \(2 \times 4 = 8\) bytes. O compilador conhece o tipo apontado e escala a conta automaticamente. A Figura 1.2 mostra por que v[i] é literalmente *(v + i):

1.5 Ponteiro para ponteiro

Se um ponteiro é uma variável, ela também tem um endereço — e podemos guardá-lo em outro ponteiro. Um int **pp é “um ponteiro para um ponteiro para int”:

// Ponteiro para ponteiro: um endereço de um endereço.
static void pointer_to_pointer(void) {
    int x = 7;
    int *p = &x;   // p aponta para x
    int **pp = &p; // pp aponta para p

    printf("x=%d  *p=%d  **pp=%d\n", x, *p, **pp); // 7 7 7
    **pp = 50;                                     // chega em x com dois saltos
    printf("x agora = %d\n", x);                   // 50
}

A Figura 1.3 mostra a corrente: pp → p → x. Cada * desfaz um salto, então **pp chega lá em x. Não é abstração gratuita: foi exatamente o ponteiro para ponteiro que tornou a remoção na lista encadeada (capítulo 6) elegante, sem caso especial para a cabeça — e é como uma função altera um ponteiro do chamador.

1.6 O ponteiro genérico void *

Todo ponteiro até aqui tinha um tipo: int *, char *. O void * é o ponteiro sem tipo — guarda qualquer endereço, mas esqueceu o que há lá. Para usá-lo, é preciso um cast que reanuncia o tipo:

// void*: guarda QUALQUER endereço, mas esquece o tipo. Para usá-lo, um cast
// recupera o tipo. É a base da estratégia Shadow Data (capítulo 7).
static void generic_pointer(void) {
    int n = 1234;
    void *anon = &n;                  // um endereço "sem tipo"
    int *back = (int *)anon;          // o cast devolve o tipo int*
    printf("via void*: %d\n", *back); // 1234
}

O void * é o que permite escrever código genérico — uma estrutura que guarda “qualquer coisa”. Foi a base da estratégia Shadow Data (capítulo 7) e é como o qsort da biblioteca padrão ordena arrays de qualquer tipo. O preço é perder a verificação do compilador: com void *, a responsabilidade pelo tipo correto passa a ser sua.

1.8 Cuidados: os ponteiros que mordem

AvisoCuidado: ponteiros inválidos

Um ponteiro guarda um endereço — e nada garante que o endereço seja válido. Os três perigos clássicos:

  • NULL: o endereço “nenhum”. Fazer *p quando p == NULL é o famoso segmentation fault. Sempre cheque antes de seguir um ponteiro que pode ser nulo.
  • Selvagem (wild): um ponteiro não inicializado aponta para lixo. Inicialize sempre — com NULL se ainda não tem para onde apontar.
  • Pendurado (dangling): aponta para uma casa que já foi liberada (free) ou que saiu de escopo. Usá-lo é comportamento indefinido. Veja mais no capítulo 2.

1.9 Pense como projetista

DicaPense como projetista

Passe um ponteiro quando a função precisa modificar o original do chamador (parâmetros de saída, como os *out do livro) ou para evitar copiar um dado grande (passar o endereço de uma struct de 1 KB custa 8 bytes, não 1 KB). Para dados pequenos que a função só lê, passar por valor (uma cópia) é mais simples e seguro. E sinalize a intenção com const: um const int *p promete “vou ler pela seta, não escrever” — um contrato que o compilador faz cumprir.

NotaNo mundo real

Ponteiros não são exclusividade do C — eles só ficam escondidos em linguagens de mais alto nível. Quando, em Java ou Python, dois nomes referenciam o mesmo objeto e mudar por um afeta o outro, são ponteiros (chamados de “referências”) por baixo. A diferença é que essas linguagens proíbem aritmética de ponteiro e gerenciam a memória por você. Tony Hoare, que inventou a referência nula em 1965, a chamou de seu “erro de um bilhão de dólares”, de tantos crashes que o NULL causou desde então — o mesmo NULL que você acabou de aprender a respeitar.

1.10 Resumo

  • A memória é uma sequência de bytes endereçados. Toda variável tem um valor e um endereço.
  • Um ponteiro é uma variável que guarda um endereço. &x dá o endereço de x; *p lê (ou escreve) o valor na casa apontada por p.
  • Por um ponteiro, uma função modifica dados do chamador — daí os parâmetros de saída e o “passar por endereço”.
  • A aritmética de ponteiros conta em elementos: p + i aponta para o i-ésimo elemento, e v[i] é só *(v + i).
  • Um ponteiro para ponteiro (**pp) guarda o endereço de um ponteiro — útil para alterar ponteiros e para evitar casos especiais (como na lista do cap. 6).
  • O void * guarda qualquer endereço sem tipo; um cast o recupera. É a base do código genérico (Shadow Data, qsort) — e abre mão da proteção do compilador.
