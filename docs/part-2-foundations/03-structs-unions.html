3  Structs e uniões: compondo tipos

Até aqui, cada variável guardava um valor: um int, um endereço. Mas o mundo é feito de coisas com várias partes — um ponto tem x e y, um nó tem um valor e um ponteiro. Para modelá-las, agrupamos campos num só tipo: a struct. Ela é o tijolo de todas as estruturas da Parte da prática — cada nó, cada container era uma struct. Aqui entendemos como o compilador a organiza na memória (com um custo escondido, o alinhamento) e conhecemos sua imagem espelhada, a união.

3.1 A struct: agrupar campos

Uma struct junta campos de tipos quaisquer sob um único nome de tipo. Você acessa os campos com . (numa variável) ou -> (por um ponteiro):

// Uma struct AGRUPA campos (de tipos quaisquer) sob um único tipo.
struct Point {
    int x;
    int y;
};

static void struct_basics(void) {
    struct Point p = {3, 4}; // inicialização por ordem
    p.y = 10;                // acesso a campo por '.'

    struct Point *pp = &p;
    pp->x = 7; // por ponteiro, '->' é açúcar para (*pp).x

    printf("Point = (%d, %d)\n", p.x, p.y);                // (7, 10)
    printf("sizeof(Point) = %zu\n", sizeof(struct Point)); // 8
}

Na memória, os campos de uma struct ficam lado a lado, na ordem em que você os declarou. Foi assim que cada Node, cada DynamicArray, cada MinHeap do livro existiu: um punhado de campos arrumados em sequência num bloco. Mas “em sequência” esconde uma sutileza.

3.2 O custo escondido: alinhamento e padding

A CPU lê a memória mais rápido quando cada tipo está num endereço alinhado ao seu tamanho — um int de 4 bytes prefere começar num endereço múltiplo de 4. Para garantir isso, o compilador insere bytes de enchimento (padding) entre os campos. O resultado surpreende: o tamanho de uma struct pode ser maior que a soma dos seus campos.

// A ORDEM dos campos importa: o alinhamento insere "buracos" (padding).
struct Padded { // má ordem
    char a;     // offset 0
    int b;      // precisa de offset múltiplo de 4 → 3 bytes de padding antes
    char c;     // offset 8; mais 3 bytes de padding no fim
}; // sizeof = 12

struct Packed { // boa ordem: do maior para o menor
    int b;      // offset 0
    char a;     // offset 4
    char c;     // offset 5; mais 2 bytes de padding no fim
}; // sizeof = 8

static void padding_demo(void) {
    printf("Padded: sizeof=%zu  (a@%zu b@%zu c@%zu)\n", sizeof(struct Padded),
           offsetof(struct Padded, a), offsetof(struct Padded, b),
           offsetof(struct Padded, c));
    printf("Packed: sizeof=%zu   (b@%zu a@%zu c@%zu)\n", sizeof(struct Packed),
           offsetof(struct Packed, b), offsetof(struct Packed, a),
           offsetof(struct Packed, c));
}

As duas structs têm os mesmos campos (um int e dois char = 6 bytes de dados), mas tamanhos diferentes. A Figura 3.1 mostra por quê: na ordem ruim, o int força 3 bytes de buraco depois do primeiro char; reordenando do maior para o menor, os buracos quase somem.

DicaTrade-off: ordem dos campos × memória

Declarar os campos do maior para o menor costuma minimizar o padding — de graça, sem mudar o código que usa a struct. Para um ponto isolado, 4 bytes a mais não importam. Mas multiplique por um milhão de nós numa estrutura, e o padding vira megabytes desperdiçados (e menos elementos por linha de cache, deixando tudo mais lento). É o tipo de detalhe que separa o código que “funciona” do código que escala.

3.3 A união: campos que se sobrepõem

A union é a imagem espelhada da struct. Numa struct, os campos ficam lado a lado (a memória é a soma deles). Numa union, os campos ficam sobrepostos — todos começam no mesmo endereço e compartilham os mesmos bytes. Por isso seu tamanho é o do maior membro, e só um membro é válido por vez.

// Uma union SOBREPÕE os campos: todos compartilham os mesmos bytes.
union Value {
    int i;
    float f;
    unsigned char bytes[4];
};

static void union_demo(void) {
    union Value v;
    v.i = 1; // escrevemos como int...

    // ...e lemos os bytes crus (little-endian em x86-64: 1 vira [1 0 0 0]).
    printf("union: sizeof=%zu  i=%d  bytes=[%u %u %u %u]\n",
           sizeof(union Value), v.i, v.bytes[0], v.bytes[1], v.bytes[2],
           v.bytes[3]);
    // Só UM membro é válido por vez: ler 'f' aqui reinterpretaria esses bits.
}

3.4 Quando uniões fazem sentido

Sozinha, uma union é perigosa: nada nela diz qual membro é o válido. O padrão útil é a união etiquetada (tagged union): uma struct que junta uma etiqueta de tipo e a union, prometendo “leia o membro que a etiqueta indica”.

struct Variant {
    enum { INT, FLOAT, TEXT } tag;  // qual membro está ativo
    union {
        int    i;
        float  f;
        char  *text;
    } as;
};

Isso modela “um valor que é um de vários tipos” gastando só o espaço do maior — em vez de carregar um campo para cada tipo possível, com todos vazios menos um. É como linguagens dinâmicas guardam variáveis, como protocolos compactam mensagens de tipos variados, e como interpretadores representam valores. A economia aparece quando há muitos desses valores e só um tipo ativo por vez.

3.5 Vendo funcionar

Point = (7, 10)
sizeof(Point) = 8
Padded: sizeof=12  (a@0 b@4 c@8)
Packed: sizeof=8   (b@0 a@4 c@5)
union: sizeof=4  i=1  bytes=[1 0 0 0]

Os deslocamentos (a@0 b@4 c@8) confirmam os buracos: entre a (byte 0) e b (byte 4) há 3 bytes de padding. E bytes=[1 0 0 0] revela a ordem little-endian do x86-64 — o byte menos significativo do int 1 vem primeiro.

3.6 Pense como projetista

DicaPense como projetista

Use uma struct para agrupar dados que andam juntos — e dê nomes claros aos campos: uma boa struct documenta o domínio. Quando o tamanho importar (muitos objetos, dados que vão para disco ou rede), ordene os campos do maior para o menor e confira com sizeof/offsetof. Use uma union (sempre etiquetada) quando um valor é um de vários tipos mutuamente exclusivos — ela troca espaço por uma promessa que você precisa cumprir à mão: ler só o membro que a etiqueta indica.

NotaNo mundo real

O layout de structs é um contrato com o mundo: formatos de arquivo binário, protocolos de rede e registros em disco dependem de saber exatamente onde cada campo cai — daí diretivas como #pragma pack para remover o padding quando o formato exige bytes coladinhos. As uniões etiquetadas são tão fundamentais que linguagens modernas as elevaram a recurso de primeira classe: são os enum com dados do Rust e do Swift, os sealed types/sealed do Java e Kotlin, e os tagged unions no coração de praticamente todo interpretador e compilador.

3.7 Resumo

  • Uma struct agrupa campos lado a lado num só tipo; acesso por . ou ->. É o tijolo de toda estrutura de dados.
  • O alinhamento faz o compilador inserir padding entre campos, então sizeof pode passar da soma dos campos. Declarar do maior para o menor minimiza o desperdício.
  • Uma union sobrepõe os membros nos mesmos bytes: tamanho do maior membro, só um válido por vez. Sozinha é perigosa; útil como união etiquetada.
  • sizeof e offsetof revelam o layout real — use-os quando a memória importar.
