4  Modularização: interface e implementação

Este é o último dos fundamentos — e o que você mais viu. Cada capítulo de código deste livro chegou em três peças: um .h, um .c e um Makefile. Não foi acaso: é assim que se organiza um programa C real. Em vez de um arquivo gigante e impossível de manter, dividimos o programa em módulos, cada um com uma interface pública (o .h) e uma implementação escondida (o .c), compilados em separado e ligados no fim. Este capítulo explica esse esqueleto.

4.1 A analogia: o cardápio e a cozinha

Num restaurante, o cardápio lista o que você pode pedir — nomes e descrições, nada de receitas. A cozinha sabe como preparar cada prato, mas fica fechada ao cliente. Você pede pelo cardápio sem saber (nem precisar saber) o que acontece lá dentro; e o chef pode trocar a receita sem mudar o cardápio.

Um módulo C funciona igual. O cabeçalho .h é o cardápio: declara os tipos e as funções que o módulo oferece. O .c é a cozinha: contém o corpo das funções e os ajudantes privados. Quem usa o módulo lê só o .h — e o autor do .c pode reescrever as entranhas à vontade, desde que respeite o cardápio.

4.2 Interface vs. implementação

Veja um módulo de conta bancária. O cabeçalho declara o quê — os tipos e as operações públicas:

/*
 * account.h — INTERFACE de um módulo de conta (o "o quê").
 *
 * Projeto:  PIESC — Estruturas de Dados e Algoritmos em C
 * Capítulo: 4 — Modularização (.h/.c)
 * Padrão:   C17 (ISO/IEC 9899:2018)
 */
#ifndef ACCOUNT_H // header guard: impede incluir este arquivo duas vezes
#define ACCOUNT_H

#include <stdbool.h> // bool

// Tipo público: saldo em CENTAVOS (inteiro) — nunca use float para dinheiro.
typedef struct {
    long balance_cents;
} Account;

// O contrato que o módulo oferece. Só isto o resto do programa enxerga.
void account_init(Account *a, long initial_cents);
void account_deposit(Account *a, long cents);
bool account_withdraw(Account *a, long cents); // false se saldo insuficiente
long account_balance(const Account *a);

#endif // ACCOUNT_H

A implementação fornece o como — os corpos das funções, num arquivo separado:

// Helper PRIVADO do módulo: 'static' o torna invisível fora deste .c.
static bool can_afford(const Account *a, long cents) {
    return a->balance_cents >= cents;
}

void account_init(Account *a, long initial_cents) {
    a->balance_cents = initial_cents;
}

void account_deposit(Account *a, long cents) {
    a->balance_cents += cents;
}

bool account_withdraw(Account *a, long cents) {
    if (!can_afford(a, cents)) {
        return false; // saldo insuficiente
    }
    a->balance_cents -= cents;
    return true;
}

long account_balance(const Account *a) {
    return a->balance_cents;
}

E quem usa o módulo inclui só a interface; não sabe nada do .c:

int main(void) {
    Account a;
    account_init(&a, 10000);   // abre com R$ 100,00 (em centavos)
    account_deposit(&a, 5000); // + R$ 50,00  -> 15000

    bool ok1 = account_withdraw(&a, 12000); // R$ 120,00 -> ok, sobra 3000
    bool ok2 = account_withdraw(&a, 99999); // saldo insuficiente -> negado

    printf("saldo: %ld centavos\n", account_balance(&a));
    printf("saque 120,00: %s | saque 999,99: %s\n", ok1 ? "ok" : "negado",
           ok2 ? "ok" : "negado");
    // can_afford(&a, 100); // ERRO de link: é static, invisível aqui
    return 0;
}

Essa separação é o que mantém um projeto sustentável. Enquanto o cardápio (.h) não muda, você reescreve a cozinha (.c) sem tocar em quem usa.

O que mora em cada arquivo.
Vai no .h (interface) Vai no .c (implementação)
Tipos públicos (struct Account) Corpos das funções
Assinaturas das funções Helpers privados (static)
Constantes/macros públicas Variáveis internas do módulo
Header guard #include dos detalhes que só ele usa

4.3 O header guard

Repare nas três linhas que cercam o account.h: #ifndef ACCOUNT_H, #define ACCOUNT_H e #endif. É o header guard, e ele resolve um problema real: um mesmo .h acaba sendo incluído mais de uma vez (porque vários arquivos o incluem, e uns incluem os outros). Sem proteção, o tipo Account seria redefinido, e o compilador recusaria. O guard diz: “se este símbolo já foi definido, pule todo o conteúdo”.

Veja o guard agir. Imagine que main.c inclui account.h direta e indiretamente (via outro cabeçalho). Ligue e desligue o guard:

viewof guard = Inputs.toggle({ value: true, label: "header guard ativo" })
{
  const inc1 = { text: '#include "account.h"   (1ª vez)', ok: true, note: "define Account ✓" };
  const inc2 = guard
    ? { text: '#include "account.h"   (2ª vez)', ok: true, note: "ACCOUNT_H já definido → conteúdo PULADO ✓" }
    : { text: '#include "account.h"   (2ª vez)', ok: false, note: "redefine Account → ❌ erro de compilação" };
  const row = (r) => `<div style="display:flex;align-items:center;gap:10px;margin:4px 0;flex-wrap:wrap">
      <code style="font:12px monospace;color:#264653;background:#f4efe9;padding:2px 6px;border-radius:4px">${r.text}</code>
      <span style="font:12px sans-serif;color:${r.ok ? "#2A9D8F" : "#E2725B"}">${r.note}</span></div>`;
  return html`<div>${row(inc1)}${row(inc2)}
    <div style="margin-top:8px;font:13px sans-serif;font-weight:600;color:${guard ? "#2A9D8F" : "#E2725B"}">
      ${guard ? "✅ compila: cada cabeçalho entra uma vez só" : "❌ não compila: «redefinição de Account»"}</div></div>`;
}

4.4 static: o privado do módulo

No account.c, a função can_afford é declarada static. Isso lhe dá ligação interna: ela existe só dentro daquele .c, invisível para o resto do programa. É o “privado” do C — um detalhe de implementação que o módulo usa, mas não expõe. Tentar chamá-la do main.c daria um erro de link: o main.o não encontra esse símbolo, porque ele não saiu do account.o. Manter privado o que é privado é o que deixa a interface pequena e o módulo fácil de evoluir.

4.5 Compilação separada

Por que .o? Porque cada .c é compilado independentemente para um arquivo objeto (.o), e só no fim o ligador (linker) junta os .os num executável. A Figura 4.1 mostra o fluxo:

O Makefile codifica exatamente esse grafo de dependências:

# Makefile — Capítulo 4: compilação separada de um projeto modular.
# Cada .c vira um .o independente; o link junta os .o num executável.

CC     ?= gcc
CFLAGS := -std=c17 -Wall -Wextra -Wpedantic -O2

# 1) Link: junta os objetos no executável final.
demo: account.o main.o
	$(CC) $(CFLAGS) -o demo account.o main.o

# 2) Compila cada .c em um .o, separadamente. Recompila se o .c OU o .h mudar.
account.o: account.c account.h
	$(CC) $(CFLAGS) -c account.c

main.o: main.c account.h
	$(CC) $(CFLAGS) -c main.c

run: demo
	./demo

format:
	clang-format -i account.h account.c main.c

clean:
	rm -f demo account.o main.o

.PHONY: run format clean
DicaIdeia central

A grande vantagem da compilação separada: ao mudar um arquivo, você recompila só ele (e religa). Num projeto com 200 arquivos, isso é a diferença entre esperar segundos e esperar minutos a cada alteração. O Makefile rastreia as dependências (note como cada .o depende do seu .c e do .h) e refaz só o necessário.

4.7 Pense como projetista

DicaPense como projetista

Projete a interface primeiro, e deixe-a mínima: exponha só o que o cliente precisa, e esconda todo o resto no .c com static. Uma interface pequena é uma promessa pequena — mais fácil de manter estável enquanto a implementação evolui. Agrupe num módulo as coisas que mudam juntas (alta coesão) e minimize o que um módulo precisa saber do outro (baixo acoplamento). Os pares da_init/da_free, bst_insert/bst_inorder de todo o livro são exatamente isto: módulos com uma interface enxuta sobre uma implementação que você pôde ler sem medo.

NotaNo mundo real

Toda biblioteca C que você já usou é um módulo desses em escala: stdio.h é a interface (você inclui e chama printf), e a implementação vem pré-compilada numa biblioteca (.a estática ou .so/.dll dinâmica) — você nunca vê o .c. É por isso que software comercial distribui cabeçalhos + biblioteca compilada: você programa contra o cardápio sem nunca entrar na cozinha. Linguagens mais novas trocam os cabeçalhos por módulos de primeira classe (os module do C++20, os crates do Rust, os pacotes de Go), mas resolvendo o mesmo problema: separar o que um componente oferece do como ele funciona.

4.8 Fim dos fundamentos

Com isto, você tem o chão completo: ponteiros (o endereço é o dado), a memória (pilha e heap, e seus fantasmas), os tipos compostos (structs e uniões) e a modularização (interface e implementação). É exatamente o ferramental que a Parte da prática usa o tempo todo — cada malloc com seu free, cada nó com seu ponteiro, cada estrutura num par .h/.c.

Se você leu os fundamentos primeiro, está pronto: vá para a Parte da prática e construa, do array dinâmico ao compactador de Huffman, as estruturas que fazem o software real funcionar — agora entendendo cada byte que elas movem.

4.8.1 Resumo

  • Um módulo separa interface (.h: o quê — tipos e assinaturas públicas) de implementação (.c: o como — corpos e helpers privados).
  • O header guard (#ifndef/#define/#endif) impede que um cabeçalho seja incluído duas vezes e cause redefinições.
  • static dá ligação interna: torna uma função ou variável privada ao seu .c — o “privado” do módulo.
  • Na compilação separada, cada .c vira um .o independente, e o ligador junta tudo. Mudou um arquivo? Recompila só ele.
  • Interface mínima, implementação escondida: alta coesão, baixo acoplamento. É o esqueleto de todo projeto C — e de todo capítulo deste livro.
